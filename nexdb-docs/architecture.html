System Architecture

Under the hood of NexDB: an in-memory schemaless store backed by aWrite-Ahead Log (WAL), B-Tree indexing, multi-client concurrency models, and migration engines.

Overview

NexDB provides ultra-fast in-memory operations while guaranteeing ACID durability by writing changes to the WAL before modifying memory state. The diagram below illustrates client APIs routing operations down into the database engine, memory mapping, index pipelines, and persistent disk writes.

Storage Engine

The storage core operates with full memory execution to maintain sub-millisecond latencies. Document records are grouped under collections (analogous to tables in SQL databases), storing schemaless JSON fields indexed by their document key strings.

The inner memory map uses thread-safe read-write locks (RwLock) for safe parallel readers while serializing write updates to avoid memory race states:

// Internal storage mapping representation
type StoreMap = HashMap<String, Arc<RwLock<Collection>>>;
//                collection          documents table

WAL and Checkpoint

Write-Ahead Log (WAL)

Before applying inserts or edits into the in-memory collections, the transaction serializer encodes changes into lines of structured JSON objects and appends them to the active log file (mydb.wal):

{"type":"Insert","id":"u2","document":{"name":"Jane","role":"admin"}}
{"type":"Update","id":"u2","document":{"name":"Jane Doe","role":"superadmin"}}
{"type":"Delete","id":"u2"}

On start, the engine reads the WAL lines from beginning to end, replaying operations to rebuild the correct memory state. This provides full durability against server failures.

Checkpointing Compaction

To avoid log accumulation on disk, the checkpoint supervisor serializes the entire in-memory database as a single consolidated state and writes it into a fresh snapshot storage file (mydb.nex). Once successfully written, the supervisor truncates the active WAL file, allowing future updates to begin on a clean slate.

Concurrency Model

NexDB coordinates multi-threaded database updates using asynchronous locks:

  • Fine-Grained Locks: Read-write locks (RwLock) guard each collection. Multiple concurrent clients can read documents in parallel without locking the database scope.
  • Serialized Writes: Writes are serialized per-collection via write locks, updating both the write-ahead log and memory atomically.
  • WAL Lock: Appending to the log requires acquiring a mutex (Mutex), guaranteeing file write safety.

Indexing

Automatic B-Tree indexing accelerates document lookups. When an index is added to a field path, NexDB builds a map linking sorted field values directly to document keys:

// B-Tree Index layout representation
type BTreeFieldIndex = BTreeMap<serde_json::Value, HashSet<String>>;
//                                field value         document keys

Indexes are updated transparently whenever documents are inserted, updated, or deleted. Queries check for index matches to reduce matching candidate records from O(n) linear searches to fast O(log n) lookups.

Query Engine

The query engine evaluates document structures against logical operators. Filters are parsed from JSON schemas and matched sequentially. If an index matches the query path, candidates are narrowed using index maps before checking remaining query operators.

Rust clients run queries natively, passing criteria filters to compile query models, which are then run against collection sets. TCP clients delegate search actions using structured JSON request filters.

Migration Engine

The migration utility translates data structures. Backups copy state into modular JSON dump formats. The restore parser parses entries, replaying inserts. Dialect generators translate collections into standard SQL commands, converting JSON types into text columns and generating insert queries automatically.
